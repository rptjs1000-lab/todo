# QA 리포트

## 요약

전반적으로 구조는 양호하나, 완료 처리 로직의 경쟁 조건(race condition), 날짜 필터링 취약점, 타입 검증 부재 등 실서비스에서 데이터 손실 또는 오동작을 일으킬 수 있는 중간~높음 심각도 이슈가 다수 발견되었다. 보안 관점에서 직접적인 XSS는 없으나 저장된 데이터에 대한 타입 검증이 없어 신뢰할 수 없는 데이터가 그대로 렌더링될 위험이 있다.

---

## 발견된 이슈

### [심각도: 높음] handleToggle — 경쟁 조건으로 인한 상태 불일치

- **파일:** `App.tsx`
- **설명:** `handleToggle` 내부에서 `todos.map()`으로 `next`를 계산한 직후, `todos.find()`로 `target`을 찾아 파티클 여부를 판단한다. `updateTodos(next)`는 비동기적으로 리렌더를 예약하므로 `target.completed`는 토글 이전 값을 기준으로 본다. 또한 클로저가 오래된 `todos`를 참조해 덮어쓸 수 있다 (stale closure).
- **수정 제안:** 함수형 업데이터를 사용하고, 파티클 트리거는 별도 플래그로 분리한다.

---

### [심각도: 높음] loadTodos — 저장된 데이터에 대한 런타임 타입 검증 없음

- **파일:** `storage.ts`
- **설명:** `localStorage`의 값은 사용자가 직접 수정하거나 다른 스크립트가 오염시킬 수 있다. 배열 여부만 확인하고 각 항목의 필드 타입을 검증하지 않아 런타임 크래시 또는 XSS 가능성이 있다.
- **수정 제안:** `isValidTodo` 타입 가드 함수를 만들어 각 항목의 필드 타입을 검증한다.

---

### [심각도: 높음] saveTodos — 저장 실패 시 사용자에게 무음 처리

- **파일:** `storage.ts`
- **설명:** `localStorage` 저장 실패(`QuotaExceededError` 등)를 `console.warn`으로만 처리한다. 사용자는 저장됐다고 생각하지만 실제로는 소실되어 신뢰 훼손으로 이어진다.
- **수정 제안:** 저장 실패 시 반환값(`boolean`)으로 호출부에 알리고, UI에서 토스트/배너로 사용자에게 경고한다.

---

### [심각도: 중간] handleAdd / handleDelete — stale closure로 인한 투두 유실 가능성

- **파일:** `App.tsx`
- **설명:** `handleAdd`와 `handleDelete` 모두 클로저가 캡처한 시점의 `todos` 스냅샷을 사용한다. 빠른 연속 입력 시 항목이 유실될 수 있다.
- **수정 제안:** `setTodos` 함수형 업데이터를 사용한다.

---

### [심각도: 중간] activeTodos 날짜 필터링 — UTC/로컬 타임존 불일치

- **파일:** `App.tsx`
- **설명:** `createdAt`은 UTC 기준 ISO 문자열이고 `getTodayISO()`는 로컬 날짜를 반환한다. 한국(UTC+9) 기준 오전 9시 이전에 생성된 투두가 날짜 불일치로 오늘 목록에서 누락될 수 있다.
- **수정 제안:** `Todo` 모델에 `createdDate` (로컬 날짜 문자열) 필드를 추가하고 필터링에 사용한다.

---

### [심각도: 중간] TodoInput — selectedCategory prop 변경 시 category 상태 미동기화

- **파일:** `TodoInput.tsx`
- **설명:** `category`의 초기값은 마운트 시 한 번만 계산된다. 이후 부모에서 `selectedCategory`가 변경되어도 입력창의 카테고리 선택이 동기화되지 않는다.
- **수정 제안:** `useEffect`로 `selectedCategory` 변경 시 `category`를 동기화한다.

---

### [심각도: 중간] ParticleEffect — DOM 직접 조작 시 이전 파티클 미정리 가능성

- **파일:** `ParticleEffect.tsx`
- **설명:** 의존성 배열에 `x, y`만 있어 값이 같으면 이펙트가 재실행되지 않는다. CSS 애니메이션이 진행 중인 파티클 노드를 즉시 제거하면 애니메이션이 중간에 잘린다.
- **수정 제안:** 파티클을 React state로 관리하거나, 이펙트 시작 시 기존 자식을 먼저 제거한다.

---

### [심각도: 낮음] 접근성(a11y) — select에 의미 있는 레이블 없음

- **파일:** `TodoInput.tsx`
- **설명:** `<select>`에 `aria-label`이 있으나 `<option>` 내용이 이모지만 있어 스크린 리더 사용자가 의미를 파악하기 어렵다.
- **수정 제안:** `<option>` 내용을 `{cat.label}`로 변경한다.

---

### [심각도: 낮음] 접근성(a11y) — CompletedModal 포커스 트랩 없음

- **파일:** `CompletedModal.tsx`
- **설명:** 모달이 열릴 때 포커스가 모달 내부로 이동하지 않고 Tab 키로 모달 외부 요소에 접근 가능하다. WCAG 2.1 모달 패턴 미준수.
- **수정 제안:** 모달 오픈 시 내부 첫 번째 포커스 가능 요소로 포커스를 이동하고 Tab 순환을 모달 내부로 제한한다.

---

### [심각도: 낮음] completedTodos — 전체 기간 데이터 로드로 인한 성능 저하

- **파일:** `App.tsx`
- **설명:** `completedTodos`는 전체 기간의 완료 항목을 가져온다. 오래된 투두가 쌓일수록 모달에 수백 개 항목이 렌더링될 수 있다.
- **수정 제안:** 오늘 완료된 항목만 표시하거나 오래된 데이터를 주기적으로 정리하는 로직을 추가한다.

## 테스트 케이스 제안

1. **연속 빠른 추가** — 100ms 간격으로 투두 3개 연속 추가 후 목록에 3개 모두 존재하는지 확인 (stale closure 검증)
2. **localStorage 손상 데이터 복구** — 악의적 문자열 직접 삽입 후 앱 로드 시 크래시 없이 빈 목록으로 시작하는지 확인
3. **localStorage 용량 초과** — 꽉 찬 상태에서 저장 시 사용자에게 오류 메시지가 표시되는지 확인
4. **자정 경계 투두** — UTC 00:00(한국 09:00) 시점에 생성된 투두가 오늘 목록에 올바르게 표시되는지 확인
5. **카테고리 필터 변경 후 입력** — 필터 변경 후 새 투두가 선택된 카테고리로 추가되는지 확인
6. **완료 토글 반복** — 빠른 완료→미완료→완료 반복 시 `completedAt` 값이 올바르게 설정/해제되는지 확인
7. **스크린 리더 테스트** — VoiceOver로 전체 플로우 순회 시 모든 요소에 의미 있는 레이블이 읽히는지 확인
8. **모달 포커스 트랩** — 완료 모달 오픈 후 Tab 반복 시 포커스가 모달 외부로 나가지 않는지 확인
9. **빈 목록 상태** — 투두 0개일 때 달성률 0%, ZeroDivisionError 없이 렌더링되는지 확인
10. **파티클 연속 완료** — 두 항목을 빠르게 연속 완료 시 파티클이 겹치거나 미정리되는 문제 없는지 확인

---

## 전체 평가

**조건부 합격**

구조적 설계는 준수하고 기본 에러 핸들링도 존재한다. 그러나 stale closure로 인한 데이터 유실 가능성(높음×2), 타입 미검증 저장 데이터의 크래시/XSS 위험(높음), 저장 실패 무음 처리(높음)의 4개 높음 심각도 이슈가 수정되기 전까지 프로덕션 배포를 권고하지 않는다. 접근성 이슈도 대응이 필요하다.
